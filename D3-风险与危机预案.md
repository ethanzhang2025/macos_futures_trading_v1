# D3 · 风险与危机预案

> 配套 D1（顶层设计）、D2（Stage A 执行）的第三层文档，专注**三件事**：
> 1. **死亡率自检** —— 避免朝错误方向烧跑道
> 2. **Plan B 资金底线** —— 让"硬扛到底"可持续且体面
> 3. **事故 SOP** —— 金融工具出问题时的响应机制

**前置阅读**：D1（顶层设计）+ D2（Stage A 执行）

---

## 1. 死亡率自检 · 最小版

### 设计取舍

**原方案**：5 个指标 + 月度正式会议
**最终方案**：**2 个指标 + 15 分钟月度 standup**

**理由**：
- 两人团队开销敏感，避免为自检而自检
- 2 个指标已足够捕捉 Stage A 最关键信号
- 自检 ≠ 终止讨论；自检 = 方向修正触发器

### 2 个核心指标

#### 指标 1：TestFlight 周新增（冷启动有效性）

| 月份 | 🟢 绿 | 🟡 黄 | 🔴 红 |
|------|------|------|------|
| M1-M3 | > 20/周 | 10-20/周 | < 10/周 |
| M4-M6 | > 50/周 | 20-50/周 | < 20/周 |
| M7-M9 | > 100/周 | 50-100/周 | < 50/周 |

**意义**：合伙人 Hunter 模式是否有效的最直接反馈。连续 4 周低于目标 50% → 改渠道或话术。

#### 指标 2：累计 Pro 付费用户数（生存底线）

| 节点 | 🟢 绿 | 🟡 黄 | 🔴 红 |
|------|------|------|------|
| M6 月底 | > 30 | 10-30 | < 10 |
| M9 月底 | > 500 | 200-500 | < 200 |

**意义**：Stage A 最终生存线。M9 < 200 = 月流水 < ¥8000 = 续不下去。

### 月度 Standup SOP（15 分钟）

**时间**：每月最后一个周五下午
**参与**：你 + 合伙人
**议程**：

1. 本月 TestFlight 新增实际值 vs 目标 → 红/黄/绿
2. 如果红：上月试了 A 方法没效果，下月试 B 方法（具体写出）
3. 本月付费数 X，累计 Y，偏差 Z
4. 结束 —— 不开大会、不写报告、不纠结

**触发 Plan B 讨论会的条件**：
- 任一指标**连续 2 个月红灯**

**Plan B 讨论会 ≠ 失败会议**：
- 审视"方向红了还是方法红了"
- 选项（详见本文档第 2 节）：砍范围 / 改渠道 / 接外包补血 / 重新考虑小额天使

## 2. Plan B · 资金底线

### 设计哲学

**"硬扛到底"与"设底线"并不矛盾** —— 底线是让硬扛**可持续**、**有体面退出空间**的保护机制。

月度开支估算（两人团队，Stage A）：**¥2.5-3 万/月**（生活费 + 基础运营）

### 🟡 应急启动线：账上余额 ≤ ¥9 万（≈ 3 个月跑道）

**触发后的动作**（不是放弃，是激活备用方案）：

- 开 Plan B 紧急讨论会
- 从以下选项中选择组合：
  - **接外包补血**：一人接活养另一人推进（生活费降到 ¥5k/月）
  - **亲友小额借款**：不稀释股权但有关系风险
  - **重新考虑 10-30 万小额天使**：换 3-5% 股份续命
  - **砍非核心开支**：退订云服务冗余、降低基础设施
  - **Stage A 发布节奏临时调整**：推迟 M7-M9 功能到 Stage B

**目的**：在还有 3 个月余力时想办法，而不是无力回天时才行动。

### 🔴 硬停线：账上余额 ≤ ¥3 万（≈ 1 个月跑道）

如果应急方案都失败，到这条线必须：

**立刻停止**：
- 不再写新功能
- 不再投入任何新花销

**用最后 1 个月做体面收场**：

- [ ] **通知用户**（给 30 天退款窗口，维护口碑）
- [ ] 退订 CTP / 云服务 / App Store 开发者账号
- [ ] **保留代码库和知识产权**（未来可能重启）
- [ ] 开最后一次结算会议，合伙人好聚好散
- [ ] 处理供应商账单和税务收尾

**原则**：不到 ¥0 才停 = 不负责任。

### 监控频率

- 每月 1 次财务对账（每月第一个周一）
- 余额进入黄色区间（¥12 万以内）→ 每周一次
- 进入红色区间（¥9 万以内）→ 每周+双方实时可见

## 3. 事故响应 SOP

### 事故分级

| 等级 | 定义 | 举例 | 响应时间 |
|------|------|------|---------|
| **P0** | 影响所有/大量用户核心功能 | 行情全断 / 集体崩溃 / **Stage B 盘中错单或无法下单** | **≤ 15 分钟** |
| **P1** | 影响部分用户核心功能 | 某品种行情错 / 部分设备崩溃 | ≤ 2 小时 |
| **P2** | 非核心功能问题 | 订阅计费错 / 画线丢失 | ≤ 24 小时 |
| **P3** | 体验问题 | 性能卡顿 / UI 小 bug | 下个版本 |

### P0 响应时间线

| 时点 | 动作 |
|------|------|
| **T+0**（发现事故） | 确认范围 → App 内 banner 推送所有用户 → 技术回滚或热修复 |
| **T+1 小时** | 官方声明（微信公众号 + 小红书 + 官网 status 页）+ 客服群"全员响应" |
| **T+24 小时** | 详细事故复盘报告（原因 / 影响 / 解决）+ 赔付方案出台 |
| **T+72 小时** | 公开改进措施 + 回信每个受影响用户 |

### 角色分工

| 职责 | 你 | 合伙人 |
|------|----|----|
| 技术响应与修复/回滚 | **主** | — |
| 用户沟通 / 客服 | 审核文案 | **主** |
| 对外声明 | 审核 | **主写** |
| 事后复盘报告 | **主** | 参与 |

**核心原则**：技术问题你处理，对外沟通合伙人主导 —— 避免你被用户情绪打断写代码。

### 赔付原则

#### Stage A（不下单）

| 问题类型 | 赔付 |
|---------|------|
| 数据损坏 | 最近备份恢复 + 订阅免费延期 30 天 |
| 订阅错扣 | 全额退款 |
| 长时间宕机（> 4 小时） | 订阅免费延期 |
| App 崩溃/性能 | Case-by-case |

#### Stage B（有下单 · 更严格）

| 问题类型 | 赔付 |
|---------|------|
| **我们代码 bug 导致错单** | **全额赔付用户损失**（需 E&O 保险覆盖） |
| 用户操作错误导致错单 | 协助但不赔付 |
| 盘中无法下单（我们的问题） | 订阅免费延期 + 个案补偿 |
| CTP 连接失败（非我们问题） | 补偿订阅延期 |

### Stage B 前置条件

- 配齐 **科技责任险 / 错误遗漏险（E&O）**
- 年费估计 ¥3-10 万
- 这是上线下单功能的**硬前置**，没保险不上线

### CTP 行情与交易断线重连 SOP（Stage A 适用行情通道 · Stage B 扩展到交易通道）

#### 正常状态监控
- **心跳机制**：每 10s 发送心跳（行情通道 + 交易通道各一）
- **状态指示**：App 右上角状态灯 —— 🟢 正常 / 🟡 重连中 / 🔴 断开 > 30s

#### 断线分级响应

| 级别 | 触发条件 | 响应动作 | 用户感知 |
|------|--------|--------|--------|
| **L1 轻度** | 心跳超时 10-30s | 静默重连 | 无 |
| **L2 中度** | 30s 无响应 / 网络切换 | Banner 提示 + 重连倒计时 | 看得到 |
| **L3 重度** | 60s 无响应 | 全屏警告 + **阻止新下单** + 进入安全模式 | 强打断 |

#### 重连策略（指数退避）

| 第 N 次 | 延迟 | 累计等待 |
|--------|-----|--------|
| 1-3 | 2s / 4s / 8s | ≤ 14s |
| 4-6 | 16s / 32s / 64s | ≤ 126s |
| 7+ | 128s 固定 | 需人工介入判断 |

#### 重连后状态校验（必须全部通过才解除安全模式）

- [ ] 查询账户资金 → 校验本地缓存 vs 服务端
- [ ] 查询持仓 → 不一致立即弹警告，人工确认后才恢复操作
- [ ] 查询未成交委托 → 对账
- [ ] 查询今日成交 → 补齐断线期间事件
- [ ] 全部对账通过 → 解除 L3 安全模式

#### Stage A vs Stage B 差异
- **Stage A**：仅行情通道。断线只影响看盘，不影响下单（本来就不下单）。SOP 相对轻
- **Stage B**：增加交易通道断线。盘中交易通道断线 = 生死级事件 → 立即 L3 + 打电话通知用户 + 公告

## 4. 对外声明原则（写进公司 DNA）

所有事故声明必须：

| 做 | 不做 |
|----|------|
| ✅ 快（越快越好） | ❌ 等查清再说 |
| ✅ 承认问题 | ❌ 推卸 / 找借口 |
| ✅ 具体赔付方案 | ❌ "深表歉意"套话 |
| ✅ 技术细节（能讲的） | ❌ 遮遮掩掩 |

### 范本（好）

> 2026-05-12 14:20-14:35，螺纹钢主连 K 线数据错误（实际价格与显示差 3 元）。
> 
> **原因**：我方行情解析模块 bug。
> **影响**：约 300 Pro 用户。
> **修复**：已于 14:40 热更新。
> **赔付**：受影响用户订阅免费延期 30 天，客服将于 24 小时内自动开通。
> **复盘报告**：72 小时内发布在 status.ourapp.com/incidents/20260512。

### 范本（坏 · 绝不使用）

> 近期部分用户反馈行情数据异常，我们已第一时间排查并修复，给您带来的不便我们深表歉意。

## 6. Plan B 备选方案库（提前想清楚）

触发应急线后可快速选用，不临时抓瞎：

### 方案 A：接外包补血
- 谁接：优先合伙人（销售能找到客户）或你（技术含量高）
- 接多久：2-3 个月，期间另一人全力推进产品
- 风险：注意力分散，产品进度延后
- 收益：每月 ¥1-3 万进账

### 方案 B：向亲友借款
- 金额：¥5-20 万
- 利息：建议 5-8% 年化（不低于银行贷款，表示正式）
- 期限：1-2 年
- 风险：关系风险 > 金钱风险，合同要写清

### 方案 C：重新考虑小额天使
- 目标金额：¥10-30 万
- 目标投资人：期货公司高管个人、金融科技领域产业资本
- 稀释：5-10% 股份
- 优势：不只是钱，还可能带来资源
- 劣势：股权不可逆

### 方案 D：砍范围 + 降开支
- 推迟 M7-M9 功能（iPad / 麦语言基础版）到 Stage B
- 换更便宜的云服务
- 合伙人改为半全职（降薪 50%，他接部分外包）

### 方案组合建议
- 应急启动线触发 → **方案 A + D 先试**
- 应急启动线 + 1 个月仍红 → **追加 B 或 C**
- 硬停线触发 → 所有方案无效，执行体面收场

## 8. 本文档维护

- **每月底 standup** 时顺手 review 一次（看有无新风险或风险等级变化）
- **重大事件后**（发布事故 / 合伙人变动 / 监管新规）立即更新
- **Stage A M3 / M6 / M9 checkpoint** 做大版本更新
- Stage B 前做深度重写（新增 CTP / 券商 / 保险相关风险）
