### 0.3 关键架构决策（在写代码前定下来）

```
决策1：最低支持macOS版本
├── 推荐：macOS 13 Ventura
├── 理由：SwiftUI成熟度 + Metal 3 + 覆盖95%用户
└── 影响：决定可用的API范围

决策2：Swift Package Manager vs Xcode Project
├── 推荐：SPM为主 + Xcode workspace
├── 理由：模块化清晰，跨平台准备（未来iOS复用）
└── 影响：项目结构和依赖管理方式

决策3：CTP封装方案
├── 推荐：方案B — C中间层封装
├── 理由：稳定可靠，避免C++ Interop的坑
├── 备选：方案C — 独立进程（如果CTP稳定性成问题再迁移）
└── 影响：整个数据/交易层架构

决策4：图表渲染方案
├── 推荐：Metal直接渲染
├── 理由：期货对性能要求极高（Tick级刷新），WebView方案不适合期货
├── 不推荐WebView：期货不像股票，Tick图需要每秒刷新数十次
└── 影响：图表引擎整体设计

决策5：数据持久化
├── 推荐：GRDB（Swift SQLite封装）
├── 理由：轻量、高性能、WAL模式支持并发读写
├── Tick存储：自定义二进制格式 + zstd压缩
└── 影响：数据层整体设计

决策6：并发模型
├── 推荐：Swift Actors（严格并发）
├── 理由：Swift 6原生支持，编译器帮你排查数据竞争
├── CTP回调线程 → Actor隔离 → UI主线程
└── 影响：所有模块间的通信方式
```

## Stage 1：技术风险验证（第3-5周）

> 目标：验证最高技术风险 — CTP能在macOS上跑通，Metal能渲染K线
> 这是整个项目的Go/No-Go决策点，如果这里过不去，后面都不用做

### 1.1 CTP封装层（第3-4周）

```
开发顺序（严格按此执行）：

第1步：C封装层
├── 将CTP的C++头文件包装为C接口
├── MdApi的C封装：
│   ├── ctp_md_create()         → 创建行情实例
│   ├── ctp_md_connect()        → 连接前置
│   ├── ctp_md_login()          → 登录
│   ├── ctp_md_subscribe()      → 订阅合约
│   └── ctp_md_set_callback()   → 设置回调函数指针
├── TraderApi的C封装：
│   ├── ctp_trader_create()     → 创建交易实例
│   ├── ctp_trader_connect()    → 连接前置
│   ├── ctp_trader_login()      → 登录
│   ├── ctp_trader_order()      → 报单
│   └── ctp_trader_set_callback() → 设置回调
├── 编译验证：C封装能链接CTP动态库
└── 目标：纯C层面能调用CTP函数，不涉及Swift

第2步：Swift封装层
├── 将C回调转换为Swift async/await
├── 设计 CTPMarketDataActor：
│   ├── func connect(front: String) async throws
│   ├── func login(brokerID:, userID:, password:) async throws
│   ├── func subscribe(instruments: [String]) async throws
│   └── var tickStream: AsyncStream<Tick>  → 行情推送流
├── 设计 CTPTraderActor：
│   ├── func connect(front: String) async throws
│   ├── func login(...) async throws
│   ├── func insertOrder(order: OrderRequest) async throws -> OrderResponse
│   ├── func cancelOrder(orderRef: String) async throws
│   ├── func queryPosition() async throws -> [Position]
│   └── func queryAccount() async throws -> AccountInfo
├── 穿透式监管信息填充（AppInfo字段）
└── 编译验证：Swift能调用C层，C层能调用CTP

第3步：SimNow连通测试
├── 连接SimNow行情前置 → 登录成功
├── 订阅 rb2501（螺纹钢）→ 收到Tick数据
├── 连接SimNow交易前置 → 登录成功
├── 查询账户资金 → 返回正确
├── 下一手螺纹钢限价单 → 委托成功
├── 撤单 → 撤单成功
├── 在控制台打印所有数据，确认解析正确
└── 目标：完整的行情+交易链路在macOS上跑通
```

**验收标准**：运行App，控制台持续打印螺纹钢的Tick数据（价格、量、时间），能下单和撤单。

**如果这一步失败**（CTP macOS SDK有兼容性问题）：
- 备选方案A：使用方案C（独立C++进程 + IPC）
- 备选方案B：自己编译开源CTP替代品（如openctp）
- 最坏情况：项目终止或转向WebSocket数据源

```

├── instrumentID: String
├── period: KLinePeriod     — 周期枚举

├── open: Decimal
├── high: Decimal
├── low: Decimal
├── close: Decimal
├── volume: Int
├── openInterest: Decimal
└── turnover: Decimal       — 成交额

合约模型：
├── instrumentID: String    — 合约代码（如rb2501）
├── instrumentName: String  — 合约名称
├── exchangeID: String      — 交易所（SHFE/DCE/CZCE/CFFEX/INE/GFEX）
├── productID: String       — 品种代码（如rb）

├── priceTick: Decimal      — 最小变动价位
├── deliveryMonth: Int      — 交割月
├── expireDate: String      — 到期日
├── longMarginRatio: Decimal  — 多头保证金率
├── shortMarginRatio: Decimal — 空头保证金率
└── isTrading: Bool         — 是否在交易

实现任务：
├── SQLite数据库Schema设计（GRDB）
├── Tick → K线实时合成引擎
│   ├── 按时间周期合成（1m/5m/15m/30m/1h/日/周/月）

│   └── 支持自定义周期（如3m/2h等）
├── 历史K线缓存（SQLite存储，启动时加载）
├── 合约信息缓存（从CTP查询后本地存储）
└── 交易日历基础实现
```

### 9.2 App Store审核要点（金融类特别注意）

```
常见被拒原因及预防：

1. 金融功能描述不清
   ├── 预防：审核备注明确说明"连接用户自有期货账户"
   └── 强调：不提供投资建议，不经手资金

   ├── 预防：提前准备中英文隐私政策页面
   └── 说明收集的数据类型和用途

3. 使用非公开API
   ├── 预防：全部使用公开API
   └── CTP动态库是用户安装的，不是App内置（如果有审核问题）

   ├── 预防：充分测试SimNow环境
   └── 非交易时段也要能正常打开App

5. 沙盒权限问题
   ├── 需要的Entitlements：
   │   ├── com.apple.security.network.client — 网络访问（CTP连接）
   │   ├── com.apple.security.files.user-selected.read-write — 数据导出
   │   └── com.apple.security.files.downloads.read-write — 可选
   └── 不需要的（避免申请）：
       ├── com.apple.security.network.server
       └── com.apple.security.cs.disable-library-validation

备选方案：如果App Store被拒
├── 独立分发（Developer ID + Notarization）
├── 许多金融类App选择独立分发
├── 好处：不受App Store沙盒限制
└── 坏处：用户安装需要额外步骤
```
