# macOS 期货交易软件 — 可行性方案与竞品分析

## 一、市场现状：为什么说这是更好的机会

| 对比维度 | 股票软件 | 期货软件 |
|---------|---------|---------|
| macOS空白程度 | 有富途、同花顺Mac版（阉割） | **几乎完全空白** |
| 用户付费意愿 | 低，习惯免费 | **高，文华收费¥720-1200+/年，用户照付** |
| 用户专业度 | 散户为主 | 专业交易者/机构为主 |
| 屏幕使用时间 | 盘中偶尔看 | **T+0全天盯盘，含夜盘到凌晨** |
| 接口标准化 | 各券商自有API，碎片化 | **CTP统一标准，一套API对接90%期货公司** |
| 牌照要求 | 交易需证券牌照 | **做软件终端不需要牌照（文华财经模式）** |

**核心结论**：期货软件的商业模式更清晰，技术路径更标准化，竞争更少，付费更好。

### 竞品核心缺陷分析

```
文华财经的痛点（直接攻击点）：
├── Mac用户只能用网页版或远程桌面 → 延迟高、体验差
├── 图表渲染陈旧 → 不支持现代图表类型(Footprint/OrderFlow)

├── 只满足最基础下单需求
├── 图表约等于无
├── 无任何编程/自定义能力
└── 完全没有Mac版

交易开拓者(TB)的痛点：
├── 价格昂贵
├── 无Mac版
├── TBL语言生态封闭
└── 用户基数小

国际软件的痛点：
├── 不支持国内CTP
├── 不支持国内期货交易所
├── 无中文
└── 合约规则不匹配
```

## 三、期货 vs 股票的核心技术差异

期货交易软件不是"把股票软件换个数据源"，机制层面有根本区别：

### 交易机制差异

| 维度 | A股股票 | 国内期货 |
|------|--------|---------|
| 交易方向 | 只能做多 | **双向交易（做多+做空）** |
| 日内交易 | T+1 禁止 | **T+0 无限次** |
| 杠杆 | 无（融资融券除外） | **5-15倍保证金杠杆** |
| 交易时段 | 9:30-15:00 | **日盘9:00-15:00 + 夜盘21:00-次日2:30** |
| 结算 | 收盘价 | **每日结算价（加权均价）** |
| 合约到期 | 无 | **有到期日，需要移仓换月** |
| 涨跌停 | 固定比例 | **品种不同，且可调整** |
| 手续费 | 按金额万分比 | **按手或按金额，开平不同** |
| 持仓 | 买入即持有 | **开仓/平仓/平今/平昨 区分** |

### 需要额外实现的核心功能

```
期货特有功能（股票软件没有的）：

1. 保证金与风控系统
   ├── 实时保证金占用计算
   ├── 可用资金 / 风险度显示
   ├── 强平线预警
   └── 各品种保证金比例表

2. 持仓管理（远比股票复杂）
   ├── 多空持仓分开显示
   ├── 平今 vs 平昨（手续费不同，上期所区分）
   ├── 逐笔浮盈 / 盯市浮盈
   ├── 持仓均价 vs 开仓均价
   └── 锁仓/解锁

3. 合约管理
   ├── 主力合约自动识别与切换
   ├── 主力连续合约（复权拼接）
   ├── 合约到期日提醒
   ├── 移仓换月辅助
   └── 合约乘数/最小变动价位表

4. 套利功能
   ├── 跨期套利（同品种不同月份）
   ├── 跨品种套利（如豆油-棕榈油）
   ├── 价差图表
   ├── 套利组合下单
   └── 蝶式/鹰式套利

5. 深度行情
   ├── 盘口深度（DOM - Depth of Market）
   ├── 分笔成交（逐笔Tick）
   ├── 成交量剖面（Volume Profile）
   ├── 订单流（Order Flow / Footprint Chart）
   └── 持仓量变化分析

6. 夜盘支持
   ├── 不同品种夜盘时段不同
   │   ├── 21:00-23:00（有色金属）
   │   ├── 21:00-23:30（天然橡胶等）
   │   └── 21:00-次日02:30（黄金/白银/原油）
   ├── 跨日K线归属（夜盘算下一交易日）
   └── 夜盘断线自动重连
```

### CTP 接入 — 最关键的技术点

CTP（综合交易平台）是上期技术开发的标准化接口，国内90%+期货公司都用它：

```
CTP架构：
├── MdApi (行情接口)
│   ├── 订阅合约行情
│   ├── 接收Tick数据推送
│   └── 深度行情(Level 2)
│
├── TraderApi (交易接口)
│   ├── 登录认证
│   ├── 报单/撤单
│   ├── 查询持仓/资金/成交
│   └── 结算确认
│
└── 技术细节：
    ├── C++ API → 需要 Swift/C++ 互操作
    ├── 官方提供 Linux/Windows/macOS 版本
    ├── macOS版从 CTP 6.6.x 开始支持 arm64
    ├── 回调模式 → 需要封装为 Swift async/await
    └── SimNow 模拟环境可免费测试
```

### Swift 封装 CTP 的方案

```
方案A：C++ Bridging Header (直接互操作)
├── Swift 5.9+ 支持 C++ Interop
├── 直接调用 CTP 的 C++ 类
├── 优点：无额外层，性能最好
└── 缺点：C++/Swift互操作仍有限制

方案B：C 中间层 (推荐)
├── 用 C 包装 CTP 的 C++ API
├── Swift 通过 C bridging 调用
├── 优点：稳定可靠，兼容性好
└── 缺点：多一层封装

方案C：独立进程 + IPC
├── CTP 跑在独立的 C++ 进程中
├── 通过 Unix Socket / gRPC 与 Swift 通信
├── 优点：隔离性好，CTP崩溃不影响UI
└── 缺点：架构复杂，延迟略高
```

## 五、图表引擎 — 对标TradingView而非文华

期货图表比股票图表要求更高，需要支持的图表类型：

```
图表类型（按优先级）：

P0 — 必须有：
├── 标准K线图 (Candlestick)
├── 分时图 (含夜盘连续)
├── 闪电图 (Tick Chart — 期货核心)
├── 成交量柱状图
└── 多周期 (1s/3s/5s/15s/1m/3m/5m/15m/30m/1h/日/周/月)

P1 — 差异化竞争力：
├── Volume Profile (成交量剖面)
├── Market Profile (市场轮廓/TPO图)
├── Footprint Chart (订单流足迹图)
├── Renko (砖形图)
├── Range Bar (等幅图)
├── Heikin Ashi (平均K线)
└── 价差图 (Spread Chart)

P2 — 专业级：
├── DOM (Depth of Market 盘口梯形图)
├── Time & Sales (逐笔成交流)
├── Cluster Chart (簇状图)
└── Cumulative Delta (累积Delta)
```

这些图表类型中，**Footprint / Volume Profile / DOM** 是文华财经完全没有的，也是TradingView的强项。如果能在macOS原生上实现这些，就是降维打击。

## 六、公式/编程语言策略

| 语言 | 用户群 | 复杂度 | 建议 |
|------|--------|--------|------|
| 麦语言 (文华) | 最大 | 极简 | **必须兼容** — 用户迁移的关键 |
| 通达信公式 | 大 | 中等 | 建议兼容 — 很多期货用户也用通达信看盘 |
| Pine Script (TV) | 增长中 | 中等 | 可选兼容 — 吸引国际用户 |
| Python API | 量化用户 | 高 | 提供SDK — 覆盖量化群体 |

### 麦语言示例（需要兼容的）

```
// 文华财经麦语言 — 双均线策略
MA5:=MA(CLOSE,5);
MA20:=MA(CLOSE,20);

// 金叉做多
CROSS(MA5,MA20),BPK;    // BPK = 买平开（平空开多）
// 死叉做空
CROSS(MA20,MA5),SPK;    // SPK = 卖平开（平多开空）
```

麦语言比通达信公式更简单，语法更少，实现一个兼容的解释器工作量更小。但需要支持文华特有的交易指令函数：

```
交易指令函数（文华特有）：
├── BK   — 买开仓
├── SK   — 卖开仓
├── BP   — 买平仓
├── SP   — 卖平仓
├── BPK  — 买平开（先平空再开多）
├── SPK  — 卖平开（先平多再开空）
├── CLOSEOUT — 全部平仓
├── STOP — 停止交易
└── 各种条件单指令
```

## 九、合规要求

**好消息：做期货交易终端软件的合规门槛比股票低得多。**

```
文华财经的运营模式（可参照）：
├── 它不是期货公司，不需要期货牌照
├── 它是"期货交易终端软件提供商"
├── 用户自己有期货账户，用软件连接期货公司的CTP
├── 软件只是一个"工具"，不碰用户资金
└── 收费模式：直接向用户收取软件使用费

需要做的：
├── 工商注册 — 经营范围含"软件开发"
├── 软件著作权登记 — 保护知识产权
├── ICP备案 — 如果有官网/服务端
├── Apple Developer — App Store上架
├── 用户协议 — 明确免责（投资风险自担）
├── 隐私政策 — 处理用户交易数据需合规
└── 数据安全 — CTP账号密码加密存储

不需要的：
├── 期货经营许可证（你不是期货公司）
├── 证券投资咨询资格（不提供投资建议）
└── 支付牌照（不经手资金）
```

### 穿透式监管（重要）

2019年文华财经因为"穿透式监管"政策被要求整改。证监会要求所有期货交易终端必须支持**穿透式监管**，即终端需要上报：
- 客户端信息（软件名称、版本、MAC地址等）
- 操作系统信息
- 用户操作记录

软件必须从一开始就支持穿透式监管接口（CTP 6.3.15+已内置支持）。

## 十一、期货 vs 股票 — 先做哪个？

| 考量 | 先做期货 | 先做股票 |
|------|---------|---------|
| 市场空白度 | 更空白 | 有竞品（阉割版） |
| 付费意愿 | 更强 | 弱 |
| 技术复杂度 | 更高（CTP+双向交易+夜盘） | 中等 |
| 用户获取 | 期货社区精准投放 | 需大面积推广 |
| 变现速度 | 更快（已验证的收费模式） | 慢（需要规模） |
| 代码复用 | 图表引擎/公式引擎可复用到股票版 | 同样可复用 |

**建议：先做期货。** 理由：

1. 更快变现，验证商业模式
2. 图表引擎和公式引擎开发完后，80%的代码可复用到股票版
3. 期货用户更专业，口碑传播效率更高
4. CTP标准化接口一次对接覆盖全行业，不需要逐个谈券商

最优策略：**图表引擎和公式引擎作为共享核心模块开发，先接期货CTP上线，再接股票行情扩展为全市场版本。**

## 十二、补充：条件单与算法单系统

CTP本身只支持最基础的限价单和市价单。高级订单类型需要**客户端本地实现**——监控行情，条件触发后自动向CTP提交委托。这是期货软件的核心竞争力之一。

```
条件单类型（必须支持）：

基础条件单：
├── 止损单 (Stop Loss) — 价格触及阈值自动平仓
├── 止盈单 (Take Profit) — 盈利达标自动平仓
├── 追踪止损 (Trailing Stop) — 止损价随浮盈动态上移
├── 触价单 — 价格到达指定位置自动下单
└── 时间条件单 — 指定时间自动执行（如尾盘平仓）

组合条件单：
├── OCO单 (One Cancels Other) — 止损止盈二选一，触发一个撤另一个
├── OTO单 (One Triggers Other) — 成交后自动挂出止损止盈
├── 括号单 (Bracket Order) — 开仓 + 止损 + 止盈 三合一
└── 阶梯单 — 分批建仓/分批平仓

算法单（专业版功能）：
├── TWAP — 时间加权均价算法，大单拆分
├── VWAP — 成交量加权均价算法
├── 冰山单 (Iceberg) — 隐藏真实数量，分批挂单
└── 狙击单 (Sniper) — 盘口出现大单时自动跟随

实现要点：
├── 所有条件单逻辑运行在本地客户端
├── 需要独立线程/Actor持续监控行情
├── 断线期间条件单暂停，重连后恢复
├── 条件单状态持久化（app重启不丢失）
└── 条件触发到委托提交延迟 < 5ms
```

## 十五、补充：Tick数据管理

期货软件的数据量远超股票，Tick级数据是核心挑战：

```
数据量估算：
├── 活跃合约约80-100个
├── 每个合约每天约 20,000-50,000 个Tick
├── 每个Tick约 200 bytes（含盘口五档）
├── 单日全市场Tick数据 ≈ 500MB-1GB
├── 一年约 250个交易日 ≈ 125-250GB
└── 如果存3年历史 ≈ 375-750GB

存储方案：
├── 实时缓存 — 内存环形缓冲区（当日Tick）
├── 当日落盘 — 收盘后压缩写入SQLite/自定义二进制格式
├── 历史存储 — 按品种/日期分文件，zstd压缩（压缩率约5:1）
├── 索引 — 时间索引，支持快速定位任意时刻
└── 清理策略 — 默认保留Tick数据6个月，K线数据永久保留

K线合成：
├── 从Tick实时合成任意周期K线
├── 秒级K线 (1s/3s/5s/10s/15s/30s)
├── 分钟级K线 (1m/3m/5m/15m/30m/60m)
├── 日/周/月K线
├── 非时间K线：Tick图、Renko、Range Bar
└── 关键：夜盘Tick归属到下一交易日
```

## 十六、补充：断线重连与异常处理

交易软件的稳定性直接关系到用户资金安全，这是生死线：

```
断线场景与处理：

CTP行情断线：
├── 检测：心跳超时（CTP内置）+ 应用层Tick间隔监控
├── 处理：自动重连（指数退避：1s → 2s → 4s → 8s → 最大30s）
├── 恢复：重新订阅全部合约，请求补齐断线期间数据
├── UI提示：状态栏显示连接状态，断线时图表暂停更新
└── 条件单：断线期间暂停监控，重连后重新激活并检查是否已触发

CTP交易断线：
├── 检测：同上
├── 处理：自动重连 + 重新登录认证
├── 恢复：查询当前持仓、委托、资金状态，与本地缓存比对
├── 关键：检查断线期间是否有成交回报遗漏
└── 报单状态同步：重连后补齐所有未确认的报单状态

App崩溃恢复：
├── 持仓数据持久化 — 实时写入本地数据库
├── 条件单持久化 — 所有活跃条件单写入磁盘
├── 未确认委托 — 启动后主动查询CTP确认状态
├── 策略状态恢复 — 程序化策略的运行上下文定期快照
└── 崩溃报告 — 集成 Apple CrashReporter，上传分析

网络切换（WiFi ↔ 有线）：
├── macOS Network Framework 监听网络变化
├── 网络切换时主动断开旧连接，建立新连接
└── 切换期间缓存本地操作，恢复后同步
```

## 十七、补充：交易日历系统

看似简单但影响面极广，做错了K线全乱：

```
交易日历的复杂性：

基本规则：
├── 周一至周五交易（法定节假日除外）

├── 日K线的"一天"= 前一晚夜盘 + 当日日盘
├── 分时图需要正确拼接夜盘和日盘
└── 周K线/月K线的边界计算

实现方式：
├── 内置交易日历表（每年更新）
├── 也可通过CTP查询交易日接口动态获取
├── 本地缓存 + 定期同步
└── 节假日信息推送通知用户
```
