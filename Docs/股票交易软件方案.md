# macOS 中文股票交易软件 — 可行性方案与开发路线

## 一、产品定位与竞品分析

| 竞品 | 平台 | 优势 | 劣势 |
|------|------|------|------|
| 通达信 | Windows 为主 | 公式系统强大，用户基数大 | macOS 体验极差（Wine/虚拟机） |
| 同花顺 | 全平台 | 功能全面 | Mac版是阉割版，卡顿 |
| 东方财富 | 全平台 | 资讯强 | Mac版功能缺失严重 |
| 富途牛牛 | 全平台 | Mac原生体验最好 | 偏港美股，A股公式系统弱 |

**机会点**：目前没有一款 macOS 原生应用能同时做到「TradingView 级别的图表」+「通达信公式兼容」。这是一个真实的空白市场。

## 二、核心技术架构

```
┌─────────────────────────────────────────────────┐
│                  macOS App (Swift)               │
├──────────┬──────────┬──────────┬────────────────┤
│  UI 层   │  图表引擎  │ 公式引擎  │   交易模块     │
│ SwiftUI  │  Metal/   │ 通达信DSL │  券商API对接   │
│ + AppKit │  Core     │  解释器   │  (CTP/恒生)   │
│          │ Graphics  │          │               │
├──────────┴──────────┴──────────┴────────────────┤
│              数据层 (Swift Actors)                │
├──────────┬──────────┬───────────────────────────┤
│ WebSocket│  REST API │    本地存储 (SQLite)       │
│ 实时行情  │  历史数据  │   公式/自选/配置           │
└──────────┴──────────┴───────────────────────────┘
```

### 技术选型建议

| 模块 | 推荐方案 | 理由 |
|------|---------|------|
| **UI框架** | SwiftUI + AppKit 混合 | SwiftUI做布局，AppKit做复杂交互（如分屏、拖拽面板） |
| **图表渲染** | Metal + Core Graphics | 高性能K线渲染，60fps缩放滚动，TradingView级别的流畅度 |
| **公式引擎** | 自研解释器 (Swift) | 解析通达信公式语法，编译为可执行的指标计算 |
| **实时数据** | WebSocket | 行情推送，低延迟 |
| **本地存储** | SQLite (via GRDB) | 历史K线缓存、用户数据 |
| **网络层** | URLSession + Combine | 原生方案，稳定可靠 |

## 三、六大核心模块详解

### 模块1：图表引擎（最核心，工作量最大）

要达到 TradingView 级别，需要实现：

```
图表引擎功能清单：
├── K线图 (日K/周K/月K/分钟K/分时图)
├── 缩放与平移 (触控板手势 + 键盘快捷键)
├── 十字光标 (跟随鼠标，显示OHLCV)
├── 多图表联动 (主图 + 多个副图)
├── 绘图工具 (趋势线/斐波那契/矩形/通道)
├── 技术指标叠加 (MA/BOLL/SAR 等主图指标)
├── 副图指标 (MACD/KDJ/RSI/VOL 等)
├── 自定义配色方案
├── 复权处理 (前复权/后复权/不复权)
└── 截图/分享
```

**实现路径有两条**：

- **路径A：纯原生 Metal 渲染（推荐）**
  - 优点：性能最好，原生体验，可定制性最强
  - 缺点：开发工作量大（约3-4个月集中开发）
  - 适合：追求极致体验，长期项目

- **路径B：混合方案 — WKWebView 嵌入 TradingView Lightweight Charts**
  - 优点：TradingView 的开源 lightweight-charts 库可直接用，开发快
  - 缺点：WebView性能天花板，与原生交互有摩擦
  - 适合：快速验证产品，MVP阶段

**建议**：先用路径B做MVP验证市场，同时并行开发路径A的原生引擎，后期切换。

### 模块2：通达信公式引擎

通达信公式本质上是一个 DSL（领域特定语言），需要构建：

```
公式引擎架构：
├── 词法分析器 (Lexer)    → 将公式文本拆分为Token
├── 语法分析器 (Parser)   → 构建AST（抽象语法树）
├── 语义分析器            → 类型检查，函数验证
├── 解释执行器            → 遍历AST，计算结果
└── 内置函数库            → 200+ 通达信标准函数
```

需要兼容的核心函数类别：

| 类别 | 示例函数 | 数量 |
|------|---------|------|
| 行情函数 | CLOSE, OPEN, HIGH, LOW, VOL, AMOUNT | ~20 |
| 数学函数 | MA, EMA, SMA, DMA, WMA | ~15 |
| 逻辑函数 | IF, IFF, CROSS, LONGCROSS, EVERY | ~10 |
| 统计函数 | HHV, LLV, COUNT, SUM, STD | ~15 |
| 引用函数 | REF, REFV, BARSLAST, HHVBARS | ~10 |
| 绘图函数 | DRAWLINE, STICKLINE, DRAWICON, DRAWTEXT | ~20 |
| 时间函数 | DATE, TIME, YEAR, MONTH, DAY, HOUR | ~10 |
| 财务函数 | FINANCE, CAPITAL, DYNAINFO | ~30 |
| 其他 | WINNER, COST, PEAK, TROUGH 等 | ~70+ |

示例 — 一个典型的通达信公式：
```
{MACD指标}
DIF:EMA(CLOSE,12)-EMA(CLOSE,26);
DEA:EMA(DIF,9);
MACD:2*(DIF-DEA),COLORSTICK;
```

### 模块3：行情数据

**数据源方案（按阶段）**：

| 阶段 | 数据源 | 成本 | 说明 |
|------|--------|------|------|
| MVP | 免费API（新浪/腾讯财经） | 免费 | 延迟约3-5秒，有频率限制 |
| 正式版 | 万得/聚宽/Tushare Pro | ¥几千-几万/年 | 专业数据，稳定可靠 |
| 交易版 | 券商Level 2接口 | 需合作 | 实盘交易必须 |

### 模块4：交易执行

**这是合规要求最严格的部分**：

- **方案A：不做交易，只做行情分析工具** → 合规门槛最低，推荐MVP阶段
- **方案B：对接券商开放API** → 如富途OpenAPI、盈透IBKR API → 中等门槛
- **方案C：自建交易通道（CTP/恒生）** → 需要证券牌照，门槛极高

### 模块5：用户界面

参考通达信/富途的核心界面：
```
核心界面：
├── 自选股列表 (支持分组、排序、多列自定义)
├── 个股详情页 (K线 + 分时 + 盘口 + 资讯)
├── 多窗口/分屏布局 (类通达信的多图同列)
├── 公式编辑器 (语法高亮 + 自动补全 + 实时预览)
├── 条件选股 (基于公式的筛选)
├── 预警系统 (价格/指标突破通知)
└── 快捷键系统 (通达信风格：数字键切换、F5/F10)
```

### 模块6：本地数据与同步
```
本地存储：
├── SQLite → 历史K线缓存（可达数GB）
├── UserDefaults → 用户偏好设置
├── 文件系统 → 自定义公式文件(.tdx兼容格式)
└── iCloud (可选) → 多设备同步自选股、公式
```

## 五、补充：A股特有机制（必须内建）

这些不是"功能"，是基础设施。漏掉任何一个，用户会直接弃用：

| 机制 | 说明 | 实现复杂度 |
|------|------|-----------|
| **涨跌停板** | 主板±10%，科创/创业板±20%，ST股±5%，北交所±30% | 需按股票类型动态计算 |
| **T+1 制度** | 当日买入次日才能卖出，持仓显示需区分 | 影响模拟交易/回测逻辑 |
| **集合竞价** | 9:15-9:25 开盘竞价，14:57-15:00 收盘竞价 | 分时图需要正确展示 |
| **复权算法** | 除权除息日精确处理，前复权/后复权价格计算 | 数学精度要求高 |
| **股票代码体系** | 60xxxx上证，00xxxx深证，30xxxx创业板，68xxxx科创板，8xxxxx北交所 | 交易所路由基础 |
| **交易时段** | 9:30-11:30 / 13:00-15:00，盘前盘后无交易 | 影响实时数据刷新策略 |

## 九、补充：数据精度

金融软件的致命细节：

```swift
// 错误 — 浮动精度问题
let price: Double = 29.10  // 实际存储 29.09999999999...
let amount = price * 100   // 结果可能是 2909.9999...

// 正确 — 使用 Decimal
let price = Decimal(string: "29.10")!
let amount = price * 100   // 精确 2910
```

- 所有价格、金额计算**必须用 `Decimal` 类型**，不能用 `Double`
- A股最小价格变动单位是 ¥0.01
- 涨跌幅百分比保留2位小数
- 成交量单位：手（100股）
- 成交金额单位：元，显示时转换为万/亿

## 十一、补充：Apple 平台特有技术点

| 事项 | 说明 |
|------|------|
| **Apple Silicon 优化** | 公式批量计算可以用 Accelerate 框架做 SIMD 加速，这是对 Windows 竞品的天然优势 |
| **App Sandbox** | Mac App Store 要求沙盒，限制文件访问/网络权限，需要正确配置 entitlements |
| **Notarization** | 非 App Store 分发必须公证，需要 Apple Developer ID (¥688/年) |
| **Hardened Runtime** | 公证要求，限制 JIT/动态库加载，影响公式引擎设计 |
| **Touch Bar** | 虽然已停产，但老款 MacBook Pro 用户仍需适配 |
| **Menu Bar** | macOS 用户期望完整的菜单栏，快捷键绑定 |
| **macOS 版本兼容** | 建议最低支持 macOS 13 (Ventura)，覆盖95%+用户 |

## 十二、补充：通达信公式完整分类

```
公式类型：
├── 1. 技术指标公式 → 画线指标 (MA/MACD/KDJ)
├── 2. 条件选股公式 → 返回布尔值，用于筛选
├── 3. 交易策略公式 → 含 BUY/SELL 信号
├── 4. 五彩K线公式  → 自定义K线颜色
└── 5. 分时主图公式 → 在分时图上叠加
```

另外还要支持：
- **公式间调用**：一个公式引用另一个公式的值
- **参数化**：`MA(CLOSE, N)` 中 N 可由用户在界面上拖动调整
- **跨周期引用**：日线公式引用周线数据
- **跨品种引用**：引用上证指数数据做对比

## 十三、补充：测试策略

金融软件的数据错误会直接导致用户亏钱，测试标准要远高于普通App：

```
测试策略：
├── 指标计算验证 → 与通达信逐一比对100+指标的计算结果
├── 复权价格验证 → 与官方数据源交叉校验
├── 边界测试     → 新股首日/ST摘帽/停牌复牌/除权日
├── 性能基准     → 5000+股票实时刷新 < 100ms
├── 内存压力     → 长时间运行不泄漏 (Instruments)
└── 断网恢复     → 网络中断后自动重连，数据不错乱
```

## 十六、技术风险与应对

| 风险 | 影响 | 应对策略 |
|------|------|---------|
| 图表性能不达标 | 用户体验差 | 早期引入Metal渲染，做性能基准测试 |

| App Store审核被拒 | 延迟上线 | 提前研究金融类App审核要求，准备好资质材料 |
| 一人开发周期过长 | 市场窗口 | 模块化设计，MVP先行，核心功能优先 |
