# A01 · 仓库骨架与架构基线

## 1. 基本信息
- 阶段：Stage A
- 时段：M1
- 优先级：P0
- 目标：建立可持续迭代的 monorepo、模块边界、编码规范、CI、feature flag、审计日志骨架。

## 2. 为什么现在做
Stage A 需要每周发版、M6 上线收费，必须先把代码组织和发布底盘搭起来。

## 3. 范围（In）
- Monorepo 目录结构（macOS App / iPad App / Shared Core / Go backend / docs）
- Swift Package 拆分：ChartCore、IndicatorCore、DataCore、JournalCore、AlertCore、ReplayCore、WorkspaceCore
- 环境配置：dev / testflight / prod
- 轻量 feature flag：UserDefaults + 远程 JSON
- 基础审计日志事件模型
- Xcode Cloud / GitHub Actions 至少一套可跑通

## 4. 不在本包范围（Out）
- 具体业务功能实现
- 复杂权限系统
- 多租户后端

## 5. 前置依赖
- 无

## 6. 输入依据
- D2 技术栈选型
- D1 产品原则 1/2/3/7
- D3 事故日志、自建审计日志要求

## 7. 必须产出
- 仓库目录初始化
- 架构说明文档 docs/architecture/stage-a-baseline.md
- .env.example / xcconfig 模板
- feature flag 服务与示例配置
- 审计日志 schema 定义
- CI 工作流文件

## 8. 建议实施步骤
- 定义顶层目录和模块边界，避免后续业务逻辑和 UI 强耦合。
- 为 macOS 与 iPad 共用逻辑建立 Shared Core 模块，UI 保持端特有层。
- 实现 feature flag 本地读取 + 远程 JSON 覆盖 + 默认兜底。
- 实现统一事件日志接口：产品事件、错误事件、订阅事件、风控事件。
- 配置 lint / format / 单元测试 / benchmark 占位任务。
- 输出 ADR（Architecture Decision Record）说明为何选 Swift Package、为何暂不做微服务。

## 9. 验收标准
- 新同事拉代码后 30 分钟内可在本地跑起空应用与空后端。
- 至少有 1 个共享模块被 macOS 与 iPad 两端同时引用。
- feature flag 可通过本地和远程两种方式切换。
- CI 对 pull request 至少执行编译 + 单测 + lint。

## 10. 禁做项
- 不要把 Stage A 还没验证的复杂需求先设计成重架构。
- 不要把 UI 状态和业务状态混写在单个 ViewModel 里。
