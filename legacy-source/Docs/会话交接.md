## 五、多窗口决策回退（Day 5，2026-04-22）

**状态**：多窗口已删，Alpha 回到单窗口。**无 OPEN 问题**。

**回退过程**（commits：`de6457c` → `9239b1d` → `b0d8866` → `eaf2372` → `78c68d2` → 最终 Day 5 回退）：
1. 单窗口关闭 crash：stack 在 `CA::Transaction::commit → pool pop → NSConcretePointerArray dealloc → _Block_release`
2. 定位 `KLineChartView` 的两个 `NSEvent.addLocalMonitorForEvents` 全局闭包强持 SwiftUI State，`onDisappear` 在 NSHostingView 随 window 关闭时不可靠 → `eaf2372` 删 keyMonitor、改 wheelMonitor 为 `NSViewRepresentable`，单窗口关闭 OK
3. 关最后一个窗口仍崩（terminate flow）→ `78c68d2` `applicationShouldTerminateAfterLastWindowClosed = false` 规避 → 关第一个窗口后主线程 hang（beach ball）
4. 证实手搓 `NSApp + 字典 + NSHostingView` 多实例在 macOS 26 SDK 下无法彻底稳住 → 回退

**保留的正面遗产**：
- `eaf2372` NSEvent monitor 修复（单窗口也受益 —— 清理了一处生命周期隐患）
- 键盘快捷键（↑↓ 切合约、1-5 切周期、Tab 切副图、ESC/Delete 绘图）**全部删除**，对齐硬约束「快捷键延后统一梳理」
- 滚轮缩放/平移 K 线走 `ScrollWheelCatcher: NSViewRepresentable`（NSView.scrollWheel），安全
- 窗口标题 `合约名 · 周期` Combine 订阅保留
- `applicationShouldTerminateAfterLastWindowClosed` 回到 `true`（关窗即退出，单窗口下这是正常行为）

**Beta 重做路径**：迁 `SwiftUI App + WindowGroup`，原生多场景由 SwiftUI runtime 管理生命周期，不踩手搓坑。

**不冲突的路径**（Alpha 范围可做）：应用内「一窗多格」分屏视图（4 宫格看多合约）—— 这是窗口**内**布局，和多窗口正交。

## 七、关键设计决策（已冻结，勿推翻）

- **Alpha 不接 CTP**：Mock 足够，真实接入放 Beta
- **Alpha 不上 Metal 图表引擎**：Canvas 满足 500-2000 bars 需求；Metal 的收益要到 L2 深度/tick tape 才显现，放 Beta
- **不为"未来换 Metal"做抽象层**：换时新起 `MetalKLineView` 平替
- **OI 副图「放弃」已翻盘**：数据源升级后 `SinaKLineBar.openInterest` 有值，已做完
- **真实主力切换（阶段 4）暂不做**：新浪 API 月份合约覆盖不完整（RB2510 空、AU2506 过期数据、IF2506 空），做了演示效果不稳
- **金融期货 parse 是近似**：非交易时段 close=last 可接受，交易时段字段顺序需进一步核实
- **Alpha 不做多窗口**（Day 5 回退决策）：手搓 NSHostingView 多实例在 macOS 26 SDK 下 pool pop over-release / teardown hang 反复发生。推 Beta 用 `SwiftUI App + WindowGroup` 一起重做。应用内「一窗多格」分屏视图不受此决策影响
- **全局 NSEvent.addLocalMonitorForEvents 禁用**：闭包对 SwiftUI State 的强持会在 NSHostingView 销毁时悬挂 → over-release。键盘功能按硬约束延后统一梳理；滚轮用 `ScrollWheelCatcher: NSViewRepresentable` 替代
