## 11 · 可能踩的坑（v15.23 batch9-69 增量）

### batch9-69 新坑

1. **跨 module 类型 import**（多次）：TrainingScenario in TradingCore 引用 KLinePeriod 必 import Shared · DailyPnL in JournalCore 必 import Foundation · 不会自动导入
2. **Linux 不进入 macOS body**（普遍）：所有 `#if canImport(SwiftUI) && os(macOS)` 包裹的 4 大新窗口代码 · Linux build 不验证 body · 仅 Mac 实机能验
3. **macOS-only 测试粒度**（batch15+40+48+57）：纯函数 / 数据层都拆出来到 Shared / TradingCore / JournalCore · UI 层 macOS-only · 这样 Linux 能测 34 测试（批次累积）
4. **@AppStorage 重启恢复 race**（batch43-44 多 tab）：必须 .onAppear 而非 init 反序列化 · @AppStorage 在 init 时未 ready
5. **NSTextView toolTip attribute**（batch42）：NSAttributedString.Key.toolTip 必须用 NSString as value · String 不会触发 hover
6. **.onContinuousHover 仅 macOS 13+**（batch68）：编译器不报错但旧系统 silent 失败 · 项目 minDeploy macOS 13 已确保
7. **Canvas labelText.measure(in:)** 需要 GraphicsContext.resolve（batch68）：Text 不能直接量测 · 必须 ctx.resolve(text).measure(...)
8. **lock 残留处理**（pts/4 另 session 持续 swift test）：等 lock 释放 / `rm -f .build/.lock` 清残留
9. **stale lock PID 检测**：`fuser .build/.lock` / 看 lock 文件内 PID 是否还在 ps
10. **Performance benchmark 跨平台**（batch57）：Linux Decimal 软件运算比 macOS native arm64 慢 5-10× · 阈值要给 2.5× safety factor
