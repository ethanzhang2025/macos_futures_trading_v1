**single 模式 OK**（v17.248 minimumThickness 修生效）· 仅 four/sixGrid/nineGrid 出问题。

**尝试过的修法**：
- **v17.245**：raw NSSplitView 嵌套替代 NSStackView · 解决撑大但默认 sizing 不均分
- **v17.246**：raw NSSplitView → NSSplitViewController + preferredThicknessFraction=1/N · grid 均分 OK 但拖不动
- **v17.247**：MonitorStackController 从 NSSplitViewController 子类改 NSViewController wrap · 没解决

**当前 hypothesis（v17.249 没解决）**：
- 可能 NSSplitViewController 嵌套时 inner NSSplitView 仍以某种方式抢 outer hit test
- 或 ChartScene SwiftUI fittingSize 在 4 个实例 + 嵌套 NSSplitView 下 forcing layout
- 或 inner NSSplitViewItem.preferredThicknessFraction 与 outer 关系冲突

**下一步排查方向**（新会话）：

1. **诊断版**：临时把 PaneContainerController 内 four 模式改成简单 NSView placeholder（不嵌入 NSSplitViewController · 用纯静态 NSView grid）· 看 outer divider 是否能拖。如果能 → 确认是 inner NSSplitView 嵌套问题。

2. **替代方案**：用 NSGridView 代替 NSSplitView 嵌套做 grid（macOS 12+ 原生）· cell 自动均分 + 不与外层 NSSplitView 冲突 hit test。

3. **激进方案**：让 PaneContainerController 不嵌入 NSSplitViewController · 直接用 SwiftUI Grid 或 LazyVGrid 渲染 N 个 ChartScene（SwiftUI level · 不走 AppKit NSSplitView）。

4. **保留方案**：临时回退到 v17.227 NSStackView 简单版（v17.245 之前）· 接受 toolbar 可能撑大风险 · 用户切机验证哪个症状更可忍。

5. **底层调查**：在 PaneContainerController 内打 NSSplitView 内部 hitTest 日志（override hitTest）· 看 outer NSSplitView divider 是否收到 mouseDown。

**memory 警示**：`feedback_补丁累积叫停` —— 同架构问题已打 5 个补丁（v17.245/246/247/248/249）· 下次必须先做架构级回顾 · 不要再叠补丁。
